# Deterministic Reporting Proposal

This document proposes a cleaner, clearer deterministic reporting model for the PodDigitizer engine. It is grounded in the current implementation in `server/routes.ts` and the training session flow in `client/src/components/tutor/StudentTopicConditioningDialog.tsx` / `client/src/components/tutor/IntroSessionDrillRunner.tsx`.

## The Deterministic Engine Explained Simply

TT's system is a response-conditioning engine. It doesn't just teach topics-it trains how students behave when difficulty appears in those topics. The engine tracks two things per topic: **Phase** (where the student breaks) and **Stability** (how reliably they hold it together).

### Phases (From TT-OS Deep Dives)
- **Clarity**: Student sees the problem clearly (vocabulary, method, reason). Breaks when they can't name terms or state steps. Prevents collapse under pressure.
- **Structured Execution**: Student executes independently. Breaks when they hesitate or forget steps. Forces action with Model → Apply → Guide.
- **Controlled Discomfort**: Student handles difficulty without rescue. Breaks when they panic or guess. Uses Boss Battles to train stability.
- **Time Pressure Stability**: Student executes under time limits. Breaks when structure collapses under urgency. Uses Timed Execution.

### Stability Levels
- **Low**: Breaks easily, needs constant help.
- **Medium**: Holds sometimes, but inconsistent.
- **High**: Reliable, but not tested under max challenge.
- **High Maintenance**: Strong and ready to advance phases.

### How the Engine Works (Non-Dev Version)
1. **Tutor runs drills**: Each drill scores the student's behavior in a topic at a specific phase/stability.
2. **Engine updates state**: Drill score moves the topic's phase/stability up or down deterministically (e.g., good score = more stable, bad score = regress).
3. **Triggers reports**: Every 2 drills = weekly report; every 8 = monthly. Reports show topic states and next moves.
4. **Parent sees outputs**: Dashboard cards show current state; reports show progress/challenges.

The engine is deterministic: Same drill inputs always produce the same state updates and outputs. No guessing.

## Current State

### What the system does today
- Tutor training drills are submitted via `/api/tutor/training-session-drill` in `server/routes.ts`.
- Each drill row stores a single topic, a phase, previous stability, and a scored training summary.
- Parent reports are auto-generated in `maybeAutoSendDeterministicReports()` when enough drill rows exist:
  - weekly reports from every 2 drill rows since the last weekly report
  - monthly reports from every 8 drill rows since the last monthly report
- `createWeeklyStructuredDataFromDrills()` and `createMonthlyStructuredDataFromDrills()` currently:
  - parse drill rows into `topicSnapshots`
  - compute `start` and `current` topic states
  - build generic status, challenge, and next-focus lines
  - store topic state rows and raw progress lines
- Topic management and training drill UI currently:
  - activate an active topic in `StudentTopicConditioningDialog.tsx`
  - start a training drill for that topic with `Start Training Drill`
  - route to `/tutor/intro-session/:studentId?mode=training&topic=...&phase=...&stability=...`
  - submit the drill through `IntroSessionDrillRunner.tsx`

### Current pain points
- Reports are built from small, chunked drill groups and may omit active topics that were not trained in that period.
- `mainTopicsCovered` is literally the topics in the current chunk, not the student’s full active topic set.
- `Next Focus` is generated by joining raw next-move phrases and can read as repetitive or mechanical.
- `bossBattleSummaryThisWeek` / `topicProgression` contains technical state lines that are valid but parent-facing and repetitive.
- The reports frequently mix multiple topics with different progress trends without a clean story.
- If a weekly chunk has duplicate topic actions, the next-focus result can repeat the same phrase.
- The monthly report copy is especially prone to sounding internal and non-actionable.
- Parent dashboard state is clean, but the report engine does not directly reuse the same parent-facing phrasing.

## Proposal: Cleaner Deterministic Reporting

### Design goals
1. Keep deterministic output anchored to actual drill data and topic state.
2. Make weekly/monthly reports topic-centered, not session- or chunk-centered.
3. Avoid raw internal phrases and instead use simple parent-friendly templates.
4. Reflect the training session update: each drill is a topic-scored unit for phase/stability.
5. Preserve the current auto-send trigger, but improve the report structure and wording.
6. Explicitly surface active topics that were not trained during the period.

### New report data shape
For each generated report, produce:
- `periodType`: `weekly` or `monthly`
- `periodStartDate`, `periodEndDate`
- `drillCount`: number of topic drills in the period
- `activeTopics`: state for all active topics as of period end
- `topicsTrained`: topics covered by drills in the period
- `topicsNotTrained`: active topics without drills this period
- `topicSummaries`: array of topic-level outcomes:
  - `topic`
  - `startState`
  - `endState`
  - `trend`: `improved` / `regressed` / `held`
  - `signal`: `breakdown`, `stable`, `mixed`, etc.
  - `nextAction`
  - `description`
- `progressSummary`: short narrative of overall progress
- `challengeSummary`: short narrative of the main challenge
- `planSummary`: short narrative of what happens next
- `nextPeriodFocus`: topic-specific action guidance
- `sourceSessionIds`

This keeps the data deterministic while letting the frontend render it with better phrasing.

### Improved report narrative structure
Use these parent-friendly sections:
- `This week we worked on:` / `This month we worked on:`
- `What progressed:`
- `What is still the biggest challenge:`
- `How the student is responding:`
- `What we are doing next:`

### Example improved weekly report template
- `This week we completed 2 training drills across 2 topics.`
- `The student made progress in Fractions by moving from Clarity (Low) to Clarity (Medium).`
- `The student is still stabilizing Structured Execution in Ratios.`
- `The main challenge was maintaining consistency under pressure in Ratios.`
- `We will continue Structured Execution drills for Ratios and preserve Clarity practice in Fractions.`
- `Algebra remains active but was not trained this week; it is still on our priority list.`

### Example improved monthly report template
- `This month we trained 4 topics and confirmed state updates in 3 of them.`
- `Algebra is now stable at Clarity (High Maintenance).`
- `Geometry is regressing from Structured Execution (Medium) to Structured Execution (Low) and needs more step-by-step reinforcement.`
- `Probability is still at Time Pressure Stability (Low); we are continuing time-pressure practice there.`
- `Next month we will prioritize drills in Geometry and Time Pressure Stability while maintaining Algebra.`

### Revised parent-facing templates with TT identity
To maintain TT's unique identity (response conditioning under uncertainty) without jargon, the templates emphasize behavioral stability and response patterns in a natural way. This keeps them clear and reassuring while distinguishing from generic tutoring.

#### Revised Weekly Template
- `This week we focused on building stronger responses in 2 topics through 2 targeted drills.`
- `Your child advanced in Fractions, moving from early clarity to more consistent understanding.`
- `Your child is building steadier responses in Ratios.`
- `The biggest challenge was staying composed when problems got harder in Ratios.`
- `Next, we'll keep strengthening responses in Ratios and maintain progress in Fractions.`
- `Algebra is still on our list but wasn't covered this week-we'll get to it soon.`

#### Revised Monthly Template
- `This month we built response strength across 4 topics, with clear updates in 3 of them.`
- `Your child now has strong, steady responses in Algebra.`
- `Your child needs more support in Geometry to stay consistent when starting problems.`
- `Your child is still learning to handle time pressure in Probability-we're working on that.`
- `Next month, we'll focus on Geometry and Probability while keeping Algebra strong.`

### Full operational structure across all phases and stability levels

These show how the system would generate parent-facing copy for ops (tutors and parents), using the revised templates. #### Clarity Phase

- **Medium Stability**: `Your child has solid basics in [topic] and is starting to apply them more independently.`
- **High Stability**: `Your child understands [topic] well and can handle it without much help.`

- **Medium Stability**: `Your child can start [topic] problems but needs reminders to stay on track.`
- **High Stability**: `Your child executes [topic] steps reliably and is building confidence.`
- **High Maintenance Stability**: `Your child performs [topic] strongly and is preparing for uncertainty.`

#### Controlled Discomfort Phase
- **Low Stability**: `Your child gets stuck when [topic] problems get harder. We're teaching persistence.`
- **Medium Stability**: `Your child tries [topic] under difficulty but needs support to push through.`
- **High Stability**: `Your child stays composed in [topic] when challenges arise.`

## Suggested reporting changes in code

### 1. Use topic-level coverage instead of chunk-only topic lists
In `createWeeklyStructuredDataFromDrills()` and `createMonthlyStructuredDataFromDrills()`:
- derive `topicsTrained` from the chunk
- capture `topicsNotTrained` by comparing against the student’s active topic list if available
- keep `mainTopicsCovered` as `topicsTrained.join(", ")`
- add `activeTopics` to reflect current topic states for the parent dashboard

### 2. Build next-focus as topic-specific actions
Replace the existing `nextFocus` builder with a deterministic mapping like:
```ts
const nextFocusByTopic = Object.entries(nextMoveByTopic)
  .map(([topic, action]) => ({ topic, action: toGerundAction(action) }))
  .filter(({ action }) => action.length > 0);

const nextFocus = nextFocusByTopic.length > 0
  ? nextFocusByTopic
      .reduce((groups, { topic, action }) => {
        groups[action] = groups[action] ? [...groups[action], topic] : [topic];
        return groups;
      }, {} as Record<string, string[]>)
      .map(([action, topics]) => `${naturalJoin(topics)}: ${action}`)
      .join("; ")
  : "Continue current drills and consolidate topic state.";
```
This prevents repeated phrases and gives parents topic-level clarity.

### 3. Keep the same `topicSnapshots` logic, but turn it into topic outcomes
Use `topicSnapshots` to compute a `topicSummaries` array:
- `trend` = compare `start` vs `current`
- `signal` = summarize `behaviorSignals`
- `description` = one sentence per topic describing progress or challenge
- `nextAction` = derived from `nextMoveByTopic[topic]`

Then build high-level narratives from those topic outcomes instead of from generic `movementLines` only.

### 4. Distinguish between `drillCount` and `topicsTrained`
Because a single training session now scores drill totals per topic, the report should emphasize drills rather than sessions.
- `drillCount` = number of drill rows used to generate the report
- `topicsTrained` = unique topics within those drill rows

This is especially important when sessions contain multiple topic drills.

### 5. Surface active-but-untrained topics
If a topic is active in the student’s conditioning map but not present in the current report chunk, add a line like:
- `Algebra is still active but was not measured this week; we will address it again soon.`

This avoids the appearance of a partial report.

### 6. Simplify internal state output
Remove or downplay `bossBattleSummaryThisWeek` and `topicProgression` in parent-facing copy, or render them as a collapsible technical appendix.
Parents should see:
- `Fractions: from Clarity (Low) to Clarity (Medium)`
- `Ratios: stable at Structured Execution (High)`
- not the raw internal format with `Started:` and `Current:` headers unless they ask for details.

## Training session UI evolution

### Current flow
- Topic cards show `Start Training Drill` for each active topic.
- The training drill modal shows prep instructions and a button that opens `/tutor/intro-session/...`.
- The drill runner scores one topic and updates the topic state immediately.

### Recommended improvements

#### A. Make the topic-focused training path explicit
- Change the topic card primary action to: `Start [topic] training drill`.
- In the modal, show a summary: `This drill will score [topic] at [phase] / [stability].`
- Explain the result: `After this drill, the topic's state and next parent report guidance may update.`

#### B. Clarify what is required before entering the drill
- Keep the prep plan, but make it a smaller, clearer checklist:
  1. Prepare X problems.
  2. Use the exact phase/stability drill sets.
  3. Record one observation per rep.
- Add a short note: `This drill updates the student's topic conditioning state directly.`

#### C. Improve post-drill feedback
After submission, show a concise summary in the tutor UI:
- `Topic updated: Fractions moved from Clarity (Low) to Clarity (Medium).`
- `Next move: Continue Clarity drills in Fractions.`
- `This update may also generate the next parent report if enough drills are complete.`

#### D. Support multi-topic sessions clearly
If a tutor runs multiple topic drills in one physical lesson, the UI should allow:
- `Finish this drill and start a new one for another topic`
- `View session drills completed so far`
- `Persist a session summary with `topic x phase/stability updates`

This reinforces that the internal training session can include multiple topic-scored drills, and the parent reports remain deterministic per drill.

## How this proposal evolves from the current system

### Now
- Reporting is grouped by fixed drill batches (2/8 rows).
- Copy is generated from aggregated state transitions and raw next-move phrases.
- Topics outside the current batch can disappear from the narrative.
- Training drill UI is functional but not explicitly topic-driven in the wording.

### Proposed
- Weekly/monthly reports become topic-centric and parent-friendly.
- `topicsTrained` and `topicsNotTrained` are explicitly surfaced.
- `nextFocus` becomes `topic: action` instead of repeated gerund phrases.
- The UI explicitly ties each drill to a topic and shows the resulting state update.
- The report generator still uses the same auto-send trigger, but the output is more usable.

## Implementation notes

### Minimal changes
- Keep the same auto-send logic in `maybeAutoSendDeterministicReports()`.
- Keep the same report insertion behavior in `insertWeeklyReport()` / `insertMonthlyReport()`.
- Refactor `createWeeklyStructuredDataFromDrills()` / `createMonthlyStructuredDataFromDrills()` to build better structured output.

### Stronger change
- If desired, replace fixed chunking with time-windowed generation (e.g. last 7 days / last 30 days) and add a `reportReady` flag.
- For now, preserve the deterministic 2/8 rule while making the produced story clearer.

### Reuse parent-facing phrases
- Reuse `PARENT_MEANING_BY_PHASE` for topic status descriptions when rendering report summaries.
- Use the same phrase set for both dashboard state and report narrative.

## Recommended field mappings

### Report fields
- `mainTopicsCovered` → `Worked this period`
- `whatImprovedThisWeek` / `strongerSkillsThisMonth` → `Progress summary`
- `studentResponsePatternThisWeek` / `responseTrendThisMonth` → `How the student responded`
- `mainMisunderstandingThisWeek` / `recurringChallengeThisMonth` → `Biggest challenge`
- `reinforcementNextWeek` / `nextMonthPriority` → `What we are focusing on next`

### Parent UI consistency
- Render the same topic state labels from `PARENT_MEANING_BY_PHASE` in parent dashboard and report commentary.
- Use `phase / stability` plus a short one-line meaning, then a clear next action.

## Conclusion

A cleaner deterministic report should do two things at once:
1. reflect the actual per-topic training drill outcomes that the updated session model now produces, and
2. translate internal conditioning state into a simple parent story.

By making the report engine topic-aware, explicit about active-but-untrained topics, and deterministic in its next-focus guidance, the system will feel more coherent and less mechanical to parents.

This proposal preserves the current auto-send behavior while making the output reliable and usable.
